**Compliant Enrichment Pipeline for Your LinkedIn Graph (Starting from export CSV)**

Your starting CSV (from  exports via manual tools or allowed workarounds like Evaboot/Surfe) typically gives you:

- Person fields: Full Name, Current Title, Company Name, Location, Industry, LinkedIn Profile URL, Seniority Level, Function/Department, sometimes basic Company Size.
- No emails/phones, no full experience, no posts/activity, no company depth.

This is **100% compliant** because the user exports it themselves. From here we build a graph (Person → Company → Niche/ICP Cluster nodes + relationship edges) using only licensed APIs and user-driven browser actions. No server-side scraping.

### Graph Node Breakdown + Sources (What Data Lives Where)

**Person Node** (core of the graph)
- Base (CSV): Name, Title, Company, Location, Industry, URL, Seniority, Function.
- Enriched Profile: Full work history, education, skills, summary.
- Contact Layer: Verified email(s), phone(s), mobile.
- Behavioral Layer: Recent posts (text/snippets), engagement patterns (likes/comments cadence), content themes, activity signals.
- Sources: APIs for static data; Chrome extension for behavior.

**Company Node** (linked via “works_at” edge)
- Base: Name, Industry, Size (from CSV).
- Enriched: Revenue range, Founded year, Funding rounds/investors, Website, Technographics (tech stack used), Growth signals.
- Sources: Dedicated company APIs.

**Niche/ICP Cluster Nodes** (dynamic, generated by your Claude agent)
- Aggregated attributes: Common titles, industries, company sizes, tech stacks, funding stages, post themes.
- Relational signals: Shared connections, mutual engagement patterns.
- Allow Custom Niche/ICP Analysis, exploration and guidance
- Support multiple models for edge analysis and weights, so the user can really explore

**Edges** (the relational magic for both goals)
- WORKS_AT, ENGAGED_WITH (post interactions), SIMILAR_TO (ICP match score), NETWORK_PATH (shortest path via mutuals or shared niches), WARM_INTRO_CANDIDATE.

### Chrome Extension for Posts, Behavior & Activity (The Missing Compliant Piece)

This is your safest way to add dynamic behavioral data without violating LinkedIn ToS.

**How it works compliantly**:
- User manually visits profiles or searches in LinkedIn/Sales Nav (no auto-paging bots).
- Extension runs **client-side only** (reads the visible DOM when the user is on the page).
- Captures: Full “About” section, visible recent posts (text + date), comments/likes the user sees, activity cadence, mutual connections shown.
- Stores everything locally (IndexedDB or local file) or lets user click “Send batch to my app” (your Claude tool).
- Never sends data to your server without explicit user action; no background automation.
- It should use claude as a running agent behind the NPM app, where LLM interaction will enhance capabilities
- It should have goals, task lists, and other methods to help guide the user through network exploration, interactions etc.
